What makes GoScript feel new

The goal is not to imitate the legacy web stack. The goal is to promote features that are radically different, visibly useful, and difficult to reproduce cleanly in JavaScript, Node.js, and the old framework ecosystem.

Go PAINT · Rendering

Direct pipeline to WebGPU

Let GoScript paint interfaces as a native render pipeline, not just HTML that eventually becomes pixels.

  • 2D UI layers
  • 3D scene layers
  • GPU-backed animation and composition
Go PAINT · Layout

Go coordinate vectors

Model components as spatial objects with explicit coordinates, scaling, and collision maps.

  • Figma-style positioning
  • Responsive vector transforms
  • Click-target collision detection
Go FAST · Packaging

npm killer workflow

Replace runtime package chaos with manifests, lockfiles, and export slices that are easy to inspect.

  • Project manifests
  • Resolved lockfiles
  • Slice-based exports
Go IRT · Architecture

Cellular micro frontend architecture

Let modules become deployable units with their own trust rules, not just components inside a page tree.

  • Module ownership
  • Local, edge, and secure remote nodes
  • Swarm topology contracts
Go FAST · Compiler

AST-first build pipeline

Keep the compiler and export flow grounded in structural understanding so AI builders can reason about code paths safely.

  • Canonical AST transforms
  • Module graph planning
  • Inspectable build output
Go IRT · Interaction

Hyper / HTTP killer

Reduce internal chatter by moving toward binary, persistent, and zero-copy-friendly communication paths where possible.

  • Realtime event routing
  • Streaming state sync
  • Less text parsing overhead
Go PAINT · UI feel

DOM bypass

Do not depend on the DOM as the center of truth for richer applications if the canvas or GPU can do the job better.

  • Canvas-native interactions
  • Overlay UI and 3D scenes
  • More game-engine-like ergonomics
Go FAST · Delivery

BO as a product slicer

Export modules, tools, pages, or app slices into deployable artifacts so one codebase can become many products.

  • exe, goe, apk, ipa, dmg
  • Inspectable export manifests
  • Module-level packaging

Old way vs future way

This is the message that usually gets people excited. The point is to show a clean break from the browser-era defaults.

Old way

HTML + CSS + JS with layout thrash, package drift, runtime bundling, and a lot of glue code between layers.

Result: a stack that can do almost anything, but only after a lot of ceremony and dependency overhead.

Future way

GoScript + Vibe + BO + Swarm with native primitives for layout, motion, packaging, and modular deployment.

Result: a language-first product system that can build websites, ERP tools, and app slices without leaning on legacy web assumptions.

Future use cases

These are the stress targets that make GoScript feel like a future language instead of a framework clone.

Scale · Spreadsheets

Planet-scale spreadsheet engine

Target millions of concurrent users, enormous sheet counts, and live updates without the app falling apart.

Paint · Documents

Figma-like document canvas

Make documents feel like spatial surfaces with blocks, layers, zoom, and interactions instead of static text pages.

Workspaces · Notion

Block docs and tables

Build block-native documents, inline tables, linked views, and file browsing in one high-performance workspace.

Future hardware

Ternary-ready execution

Keep the architecture open to ternary, multi-valued, and future quantum-capable execution models.
